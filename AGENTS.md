# AGENTS.md

## Project

Duty RP Control System — внутренняя база учёта RP-группировки «Долг».

Это не публичная лорная вики, а рабочая система учёта:
- сталкеров;
- групп сталкеров;
- квартир и квартирантов;
- заданий;
- продаж;
- покупок;
- состава группировки «Долг»;
- журналов действий.

Публичный язык интерфейса: русский.

## Stack

- Next.js App Router
- TypeScript
- Tailwind CSS
- Prisma
- Supabase Postgres в будущем
- shadcn/ui в будущем

## Stalker profile

Профиль сталкера должен содержать:

- ФИО
- Позывной
- Дата рождения
- Принадлежность
- Особенности внешности
- Заметки
- Статус профиля
- Отметка активного или просроченного задания

Статусы профиля:

- Активен
- Архив

Принадлежность:

- Одиночка
- Долг
- Свобода
- Гопник
- Бандит
- Наёмник
- Военный
- Чистое Небо

Профили можно редактировать, архивировать, возвращать из архива и удалять с подтверждением.

В профиле сталкера должны быть вкладки или блоки записей:

- Задания
- Продажи
- Покупки
- Нарушения

Запись, созданная в профиле сталкера, должна быть привязана к этому сталкерскому профилю.

Все записи из профилей должны также отображаться в соответствующих общих журналах:

- задания — в журнале заданий;
- продажи — в журнале продаж;
- покупки — в журнале покупок;
- нарушения — в журнале нарушений.

## Stalker groups

Группа сталкеров должна содержать:

- Название
- Состав
- Заметки
- Статус

Статусы группы:

- Активна
- Архив

Роли участников группы:

- Лидер
- Участник
- Ручной ввод

Если выбрана роль «Ручной ввод», должно быть отдельное текстовое поле для названия роли.

Группы можно редактировать, архивировать, возвращать из архива и удалять с подтверждением. Удаление группы не удаляет профили участников.

## Apartments and tenants

Раздел квартир — рабочий раздел учёта проживания.

Это внутренняя строгая база учёта группировки «Долг», а не публичная гостиничная система.

Во вкладке или разделе «Квартиры» должны быть три квартиры.

Квартира должна содержать:

- Название или номер
- Статус
- Жильцы
- Оплата проживания
- Заметки

В общем списке квартир каждая квартира должна показывать:

- название или номер квартиры;
- статус занятости;
- количество жильцов;
- статус оплаты проживания;
- краткую информацию об оплате;
- заметки, если есть.

Статусы квартиры:

- Свободна
- Занята

При открытии конкретной квартиры пользователь должен видеть профиль квартиры.

В профиле квартиры должны быть два основных блока:

- Жильцы
- Оплата

Блок «Жильцы» должен показывать всех, кто записан в квартиру.

В жильцы можно добавить:

- отдельные сталкерские профили;
- сразу сталкерскую группу.

Если добавляется сталкерская группа, в жильцы квартиры должен импортироваться состав этой группы.

В списке жильцов должны отображаться:

- аватар сталкерского профиля, если он есть или если уже предусмотрен визуальный плейсхолдер;
- ФИО / Позывной;
- ФИО;
- принадлежность или краткая служебная информация, если это уже есть в данных профиля.

Блок «Оплата» должен реализовать механику принятия и отслеживания оплат за проживание.

Запись об оплате должна содержать:

- дату оплаты;
- сумму оплаты;
- до какого числа оплачено;
- заметки.

У квартиры должен рассчитываться статус оплаты.

Статус оплаты должен отображаться бейджем:

- в профиле квартиры;
- в общем списке квартир.

Статусы оплаты:

- Оплачено
- Истекает
- Просрочено
- Нет оплаты

Логика статусов оплаты:

- если оплат нет — «Нет оплаты»;
- если дата «оплачено до» уже прошла — «Просрочено»;
- если срок оплаты скоро заканчивается — «Истекает»;
- если срок оплаты ещё актуален — «Оплачено».

Порог для статуса «Истекает» — 3 дня до окончания оплаты, если другое не указано отдельно.

Заселение должно поддерживать три сценария:

1. Добавить сразу группой
2. Добавить группу и конкретные сталкерские профили
3. Добавить конкретные сталкерские профили

На текущей стадии данные квартир, жильцов и оплат должны храниться локально через localStorage.

## Tasks

Задания — рабочий подраздел внутри раздела «Журналы».

Задание должно содержать:

- Дата выдачи
- Выполнить до
- Описание задания
- Награда
- Заметки
- Кто выдал
- Кто принял выполнение
- Кому выдано
- Статус выполнения

Типы заданий не нужны.

Задание можно выдать только:

- сталкеру;
- группе сталкеров.

Задание можно создать:

- из профиля сталкера;
- из журнала заданий.

Если задание создаётся из профиля, оно привязывается к этому профилю.

Если задание создаётся из журнала, исполнитель указывается одним из способов:

- выбор существующего сталкерского профиля;
- ручной ввод имени или позывного, если профиля нет в базе.

Задание также может быть связано с группой сталкеров, если это не противоречит текущей модели данных.

Задания членам «Долга» не выдаются.

## Trade

Продажи и покупки считаются с точки зрения группировки «Долг».

Продажа — «Долг» продаёт предмет сталкеру, группе или вручную указанному покупателю.

Покупка — «Долг» покупает предмет у сталкера, группы или вручную указанного продавца.

Операция должна содержать:

- тип операции;
- кому или от кого операция;
- список предметов;
- количество;
- цена;
- общая сумма;
- кто оформил;
- заметки;
- дата операции.

Продажи и покупки можно создать:

- из профиля сталкера;
- из журнала продаж или покупок.

Если операция создаётся из профиля, она привязывается к этому профилю.

Если операция создаётся из журнала, участник операции указывается одним из способов:

- выбор существующего сталкерского профиля;
- выбор группы сталкеров;
- ручной ввод имени или позывного, если профиля нет в базе.

## Violations

Нарушение можно создать:

- из профиля сталкера;
- из журнала нарушений.

Если нарушение создаётся из профиля, оно привязывается к этому профилю.

Если нарушение создаётся из журнала, нарушитель указывается одним из способов:

- выбор существующего сталкерского профиля;
- ручной ввод имени или позывного, если профиля нет в базе.

Нарушение должно содержать:

- описание;
- дату;
- кто оформил;
- заметки.

## Duty members

Раздел состава группировки хранит профили членов «Долга».

Профиль члена группировки похож на профиль сталкера, но дополнительно содержит:

- звание;
- должность;
- подразделение;
- статус службы.

## Journals

Журналы в проекте имеют две роли:

- рабочие журналы записей;
- технический журнал важных действий системы.

Рабочие журналы записей:

- журнал заданий;
- журнал продаж;
- журнал покупок;
- журнал нарушений.

В рабочих журналах должны отображаться все записи из профилей:

- все задания всех сталкеров — в журнале заданий;
- все продажи всем сталкерам — в журнале продаж;
- все покупки у всех сталкеров — в журнале покупок;
- все нарушения всех сталкеров — в журнале нарушений.

Запись можно создать не только из профиля сталкера, но и из соответствующего журнала.

Если запись создаётся из журнала, участник указывается одним из способов:

- выбор существующего сталкерского профиля;
- ручной ввод имени или позывного, если профиля нет в базе.

Технический журнал должен фиксировать важные действия:

- создан профиль сталкера;
- изменён профиль сталкера;
- профиль перенесён в архив;
- создана группа;
- сталкер добавлен в группу;
- создана квартира;
- выполнено заселение;
- принята оплата;
- выдано задание;
- задание выполнено;
- оформлена продажа;
- оформлена покупка;
- оформлено нарушение;
- создан профиль члена «Долга».

## UI style

Интерфейс должен выглядеть как современная внутренняя система учёта военизированной группировки «Долг»:

- тёмная тема;
- строгий военный стиль;
- аккуратные панели и карточки;
- хорошая читаемая типографика;
- красные, бордовые, графитовые, металлические или приглушённые акценты;
- понятная визуальная иерархия;
- качественные карточки профилей и списков;
- аккуратные таблицы и рабочие блоки;
- интерфейс должен быть не декоративным, а рабочим, но при этом визуально качественным.

При проектировании новых экранов нужно отдавать приоритет качественному UX, читаемым карточкам, понятным действиям и визуальной иерархии, а не буквальной имитации терминала или старого ПДА.

Не делать яркий киберпанк, кислотные цвета или перегруженный интерфейс. Сохранять максимальную читаемость, русский язык интерфейса и концепцию внутренней базы группировки «Долг».

## Development rules

- Не добавлять новые зависимости без отдельного задания.
- Не подключать базу данных без отдельного задания.
- Не подключать авторизацию без отдельного задания.
- Не создавать лишние разделы.
- Все тексты интерфейса писать на русском.
- Компоненты делать переиспользуемыми.
- Все списки и журналы должны иметь пагинацию.
- Использовать TypeScript.
- После изменений проверять проект командой npm run lint.
