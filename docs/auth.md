# Служебный доступ

Система закрыта служебным входом. Пользователи работают через Supabase Auth и запись
`AccessUser`, связанную с `auth.users` через `AccessUser.authUserId`.

## Создание пользователей состава

Штатных пользователей создают в разделе `Состав` через действие `Добавить пользователя`.

Создание выполняется сервером:

1. создаётся пользователь Supabase Auth;
2. создаётся запись `AccessUser`;
3. создаётся связанный профиль состава;
4. профиль появляется в подразделе `Профили состава`.

Технический адрес `AccessUser.authEmail` формируется и хранится отдельно от отображаемого
логина. Пароль `AccessUser.password` больше не используется приложением и не должен
запрашиваться браузером.

## Вход

На странице `/login` пользователь может ввести:

- Supabase Auth email (`AccessUser.authEmail`);
- внутренний `login`;
- отображаемое имя `displayName`, если для проекта включён pre-login lookup.

В серверной Vercel-версии `/api/auth/login` находит активного `AccessUser` по внутреннему
логину и выполняет вход через `AccessUser.authEmail`.

В статической GitHub Pages-версии серверных route handlers нет. Поэтому браузер:

1. очищает локальное временное состояние Supabase Auth перед новой попыткой входа;
2. резолвит введённый идентификатор в `AccessUser.authEmail`;
3. вызывает `supabase.auth.signInWithPassword({ email: authEmail, password })`;
4. после успешного входа повторно читает профиль `AccessUser` по `auth.uid()`;
5. если профиль не найден или неактивен, локальная сессия очищается.

Для входа по `login`/`displayName` на GitHub Pages нужно применить SQL:

```text
supabase/prelogin-auth-lookup.sql
```

Он создаёт минимальную RPC-функцию `public.resolve_access_user_auth_email(text)`.
Функция доступна `anon`, но возвращает только `authEmail` активной записи и не выдаёт
прямой `SELECT` по таблице `AccessUser`.

Если этот SQL не применён, на GitHub Pages гарантированно работает только вход по
реальному Supabase Auth email.

## Уровни допуска

В интерфейсе используются два уровня:

- `Офицерский допуск`;
- `Базовый допуск`.

Роль `manager` в разделе `Состав` управленческих прав не даёт и считается базовым
допуском.

## Безопасность

- Браузер не использует `service_role`, secret key, `DATABASE_URL` или `DIRECT_URL`.
- Браузер не читает и не отправляет `AccessUser.password`.
- `authEmail` не считается секретом, но pre-login lookup должен возвращать только
минимально необходимое значение для входа.
- Роль и активность пользователя проверяются только через `auth.uid()` и серверные
данные `AccessUser`.
